Customization

How to adapt Dittonomics to your own Codex setup

Overview

Dittonomics is designed to be customized without losing the architecture.

The main rule is:

  • keep generic user-level defaults generic
  • keep project-specific behavior in repo overlays
  • keep provenance available but separate from active operating guidance
  • keep writing voice optional and explicit rather than buried in prompts

Codex Configuration Layers

Layer Use
.codex/ in this repo forkable starter bundle for the user-level Codex layer
AGENTS.md session constitution and stable working rules
AGENTS.override.md repo-specific overlay
.agents/skills/ repo-local extensions and domain skills
~/.codex/skills user-level workflow skills
~/.codex/agents custom subagents
~/voice optional prose-style hierarchy for writing and review
workflow references active domain, journal, migration, and exploration guidance

Constitution Layer

In the Codex adaptation, the constitution layer should stay lean.

Good material for AGENTS.md:

  • role philosophy
  • repo path conventions
  • language defaults
  • verification expectations
  • escalation rules
  • high-level voice policy if the repo wants one

Bad material for AGENTS.md:

  • long tables of journal profiles
  • full templates
  • provenance dumps
  • detailed tutorials that belong in separate references
  • private voice examples that should stay personal

Workflow Kernel Setup

The public starter now includes a stronger clo-workflow kernel.

The recommended minimal public-safe setup is:

  • ~/.codex/AGENTS.md
  • ~/.codex/config.toml
  • ~/.codex/skills/clo-workflow/
  • ~/.codex/skills/clo-write/
  • ~/.codex/agents/

Example config fragment

model = "gpt-5.4"
model_reasoning_effort = "high"

[features]
multi_agent = true

[agents]
max_threads = 6
max_depth = 1

[plugins."github@openai-curated"]
enabled = true

[plugins."notion@openai-curated"]
enabled = true

Treat this as a starting point, not a mandatory preset. Your own machine-specific MCP servers, plugins, and sandbox preferences belong in your local config.

Voice Setup

Voice is optional. Use it only if you want writing workflows to preserve a recognizable prose register across sessions.

Recommended public structure

~/voice/
  core_voice.md
  voice_examples.md
  econ_paper_register.md
  style_hierarchy.md
  collaboration_register.md          (optional)
  journal_tightening_register.md     (optional)

What each file does

  • core_voice.md: non-negotiable prose traits
  • voice_examples.md: short examples of good sentences or paragraph moves
  • econ_paper_register.md: field-appropriate norms for evidence, restraint, and cadence
  • style_hierarchy.md: precedence rules when files disagree
  • optional overlays: collaboration or journal-specific tightening layers

Public-safe rule

The public repo should document the structure and precedence, but should not publish a personal private voice file verbatim unless you want that voice to become part of the starter itself.

That is why Dittonomics ships templates in voice/ rather than your live private files.

Skills Layer

Codex skills should expose workflows explicitly and use clear reference loading rules.

The updated public stack now makes these points clearer:

  • clo-workflow is the kernel
  • other clo-* skills route through that kernel rather than redefining it
  • creators create and critics critique
  • the parent orchestrator owns persistence unless it explicitly delegates a write target

Writing Layer

The public clo-write starter now reflects a more mature writing workflow:

  • front-end work can be governed by a specialist intro skill
  • writing uses section contracts and paragraph planning
  • voice is treated as a style layer
  • proofread persistence belongs to the parent orchestrator

Repo Overlay Pattern

For a project repo:

  • put stable repo rules in AGENTS.md
  • put project-specific deviations in AGENTS.override.md
  • put repo-only skills in .agents/skills/
  • create explorations/ when the project needs a prototype sandbox
  • keep repo-local writing voice rules local if they should not become public defaults

Practical Customization Strategy

Start with:

  1. a lean AGENTS.md
  2. a small repo overlay
  3. a stable skill naming convention
  4. only the subagents you actively use
  5. an optional voice layer only if writing consistency is a real need

Add more structure only when it prevents repeated failure, not because the upstream project had another file.
