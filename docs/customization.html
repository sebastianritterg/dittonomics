Customization

How to install Dittonomics, set up your own voice, and adapt it for a team

What To Customize

There are four customization layers:

Layer Customize this when Where
public starter you want to change what colleagues copy by default repo .codex/
local install you want your own always-on user setup ~/.codex/, ~/.agents/skills
project repo one project needs special rules AGENTS.md, AGENTS.override.md, .agents/skills/
voice you want prose continuity across sessions ~/voice/

Recommended Install Pattern

Fork-first option

If you want your own maintained version of Dittonomics for yourself or your team, start by forking the public repo:

  • https://sebastianritterg.github.io/dittonomics/

Then:

  1. fork the GitHub repository
  2. clone your fork
  3. adapt the public .codex/ starter bundle
  4. publish your own docs site if you want a team-specific manual

Use this route when you want to:

  • keep extending the system over time
  • document your team’s conventions
  • maintain your own website or starter defaults

If you only want the local tooling and do not need your own public variant, you can skip the fork and just copy the starter bundle into ~/.codex/.

1. Start from the public starter

Copy or adapt:

.codex/

into:

~/.codex/

Minimum pieces:

  • AGENTS.md
  • skills/
  • agents/
  • workflow references
  • utility scripts

2. Keep machine-specific config local

Your own local machine settings belong in:

~/.codex/config.toml

Examples:

  • MCP servers
  • plugin preferences
  • trust settings
  • sandbox preferences

3. Keep project-specific rules in the repo

Use:

  • AGENTS.md
  • AGENTS.override.md
  • .agents/skills/

Do not overload the user-level install with rules that only belong to one paper.

How To Build Your Own Voice

The public repo includes templates in voice.

Copy the templates privately

Copy these into ~/voice/:

  • core_voice.template.md
  • voice_examples.template.md
  • econ_paper_register.template.md
  • style_hierarchy.template.md

Optional:

  • collaboration_register.template.md
  • journal_tightening_register.template.md

Then rename them without .template.

What each file does

File Purpose
core_voice.md your non-negotiable prose traits
voice_examples.md examples of sentences, paragraph moves, cadence
econ_paper_register.md field-appropriate restraint and evidence norms
style_hierarchy.md precedence rules when files conflict
collaboration_register.md optional overlay for joint-paper mode
journal_tightening_register.md optional overlay for tighter journal-facing prose

The voice rule

Your voice layer should shape:

  • diction
  • cadence
  • paragraph texture
  • compression level

It should not override:

  • evidence discipline
  • causal restraint
  • intro architecture
  • repo-local terminology rules

A good minimal voice setup

If you want the smallest useful version, start with:

  • core_voice.md
  • voice_examples.md
  • style_hierarchy.md

You can add the others later.

Team Setup

If colleagues will use Dittonomics too, use this split:

Public or shared

  • website manual
  • public starter bundle
  • project AGENTS.md
  • generic workflow rules

Private

  • personal voice files
  • machine-specific config
  • private examples you do not want to publish

Project-specific but shared

  • repo-local overlays
  • project terminology rules
  • team conventions for directories and outputs

Recommended Repo Additions

For working research repos, I recommend:

AGENTS.md
AGENTS.override.md            (optional)
.agents/skills/               (optional)
explorations/
.codex-state/                 (local, usually gitignored)

Also add:

.codex-state/

to .gitignore unless you explicitly want checkpoint files committed.

How To Extend Dittonomics Later

Because this is a Ditto-style system, you should assume it will grow.

Good ways to extend it:

  • add a new skill when a workflow recurs
  • add a new utility when a maintenance action repeats
  • add a repo-local skill when only one project needs it
  • add a new website section when colleagues keep asking the same question

Bad ways to extend it:

  • embedding private voice directly into the public starter
  • adding machine-specific paths to public files
  • hiding important behavior in undocumented scripts

Colleague Onboarding Checklist

  1. read Quick Start
  2. read User Guide
  3. bookmark Reference
  4. copy the public starter into ~/.codex/
  5. optionally set up ~/voice/
  6. learn the daily loop:
    • resume-context
    • workflow skill
    • checkpoint
