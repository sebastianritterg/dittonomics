Architecture

How the local install, starter bundle, checkpoints, and website fit together

System Layers

Dittonomics has four layers that matter in practice.

Layer Where it lives What it does
Website manual guide/ and docs/ teaches humans how the system works
Public starter bundle .codex/ in this repo gives a generic forkable install
Live local install ~/.codex/, ~/.agents/skills, ~/voice/ runs your actual working system
Project repo overlay AGENTS.md, AGENTS.override.md, .agents/skills/ makes behavior repo-specific

The public website explains the system. The starter bundle ships the generic version. Your local install is the live machine. The repo overlay makes each project real.

The Orchestrator Contract

The main Codex session is the orchestrator.

That means it:

  • reads repo guidance first
  • decides which skill to invoke
  • dispatches subagents only for bounded specialist work
  • resolves paths before delegation
  • owns persistence unless it explicitly delegates a write target

This is why Dittonomics does not install a separate orchestrator agent.

Worker-Critic Architecture

The system keeps creation and evaluation separate.

Mode Creation layer Review layer
discovery librarian or explorer paired critic
strategy strategist strategist_critic
execution coder or data_engineer coder_critic
writing writer writer_critic
peer review editor and referees blind review posture
submission verifier final mechanical gate

This separation is the main reason the workflow scales beyond one-off prompting.

Explicit Skills Instead Of Hidden Hooks

Upstream Clo-Author uses hidden Claude mechanics in some places. Dittonomics converts those ideas into explicit, visible surfaces.

Hidden upstream behavior Dittonomics surface
slash commands $clo-* skills
hidden hooks explicit research tools
implicit context survival checkpoint and resume-context
buried writing habits explicit ~/voice files

The rule is simple:

if behavior matters, it should have a visible home

Local State And Checkpoints

The new state layer lives in:

.codex-state/

It is local repo memory, not canonical research output.

Default structure

.codex-state/
  checkpoints/
    2026-04-22T18-30-00Z_end-of-day.json
    2026-04-22T18-30-00Z_end-of-day.md
  latest.json
  latest.md
  resume-context.md

Why both JSON and Markdown exist

The JSON files are:

  • stable
  • machine-readable
  • good for scripted recovery later

The Markdown files are:

  • easy to read
  • easy to share with a colleague
  • good for quick manual inspection

Recommended policy

For working research repos:

  • keep .codex-state/ local
  • add it to .gitignore unless you explicitly want to share checkpoints
  • treat it as operational memory

How Resume Works

Resume is not magic memory. It is a disciplined reconstruction step.

It combines:

  • the latest checkpoint
  • current git state
  • recent commits
  • current file existence

That is enough to restart a session cleanly without pretending the system has silent hidden memory.

Voice As A Layer

The voice layer is modular and optional.

It lives in ~/voice/ and sits below repo guidance and front-end structure rules.

That means:

  • voice shapes prose
  • voice does not replace evidence
  • voice does not replace intro architecture
  • voice does not outrank repo-specific rules

Public Repo, Local Install, And Website

This repo is intentionally both:

  • a manual
  • a starter kit

That matters for colleagues.

Someone can:

  1. read the website
  2. copy the starter bundle
  3. install it locally
  4. add a repo overlay
  5. start using the same workflow with minimal guesswork

The Ditto Expansion Rule

As you add more capabilities later, keep the architecture extensible.

Good additions:

  • new skills with clear boundaries
  • new utilities with explicit names
  • new public docs sections
  • new optional overlays

Bad additions:

  • hidden behavior nobody can explain
  • one-off hacks that bypass the skill system
  • private defaults silently turning into public starter rules
