Architecture

How Dittonomics keeps the upstream pipeline logic while changing the implementation layer

System Overview

The upstream architecture Dittonomics builds from is powerful because it separates:

  • project constitution
  • always-needed workflow rules
  • on-demand references
  • on-demand workflows
  • specialist agents
  • infrastructure behavior

The Codex adaptation keeps that shape, but translates the layers into Codex-native surfaces.

Layer Codex-native location Loaded when
Session constitution AGENTS.md every session in the repo
Repo overlay AGENTS.override.md every session when present
Repo-local skills .agents/skills/ on demand or by explicit mention
Forkable starter bundle .codex/ in this repo when a user copies or adapts the public starter
User skills ~/.codex/skills on demand or implicit by policy
Custom subagents ~/.codex/agents when the main session delegates
Explicit utilities and config scripts, references, config.toml when invoked

The important design difference is that Codex should not be documented as if hidden Claude hooks still exist. If behavior matters, it needs an explicit home.

The public repo therefore carries two distinct layers:

  • a forkable .codex/ starter kit for users who want an upstream-style distribution folder
  • repo-native AGENTS.md, AGENTS.override.md, and .agents/skills/ patterns for actual project customization

Agent Workflow Diagram

Agents by Phase

The structure remains worker -> critic:

  • Discovery: librarian -> librarian_critic, explorer -> explorer_critic
  • Strategy: strategist -> strategist_critic
  • Execution: data_engineer or coder or writer -> paired critics
  • Peer Review: editor plus two blind referees
  • Submission: verifier as final gate

The Orchestrator

The main Codex session is the orchestrator.

That means:

  • it reads repo guidance first
  • it decides when to invoke a skill
  • it delegates only bounded specialist work
  • it integrates outputs and applies quality gates

The architecture explicitly rejects the idea of installing a separate orchestrator agent when the main Codex session already has that responsibility.

The Worker-Critic Loop

The point is not bureaucracy. It is to keep creation and evaluation separate long enough to expose weak arguments before they become expensive.

Plan-First Workflow

The architecture still favors plan-first behavior when the task is consequential:

  • inspect the repo
  • identify the real artifact to produce
  • choose the right workflow phase
  • decide whether delegation helps
  • verify before claiming done

The Codex version should express this in AGENTS.md, skills, and repo overlays rather than assuming a Claude-only planning protocol exists in the background.

Parallel Agents

Parallel delegation still helps, but only when the work is:

  • concrete
  • bounded
  • materially useful
  • non-overlapping in write scope

The parent session should keep urgent critical-path work local and push only sidecar specialist tasks outward.

Scoring Protocol

The quality ladder remains:

  • 80: good enough to commit
  • 90: good enough to merge
  • 95: good enough to submit

The numbers matter because they make “review” operational rather than rhetorical.

Additional Workflows

Talks

Talk creation remains parallel to peer review, not a final afterthought.

Replication-first coding

The Codex adaptation should still encourage:

  • stable file naming
  • reproducible outputs
  • explicit serialization of reusable intermediates
  • review before promotion into manuscript assets

Exploration-first side work

explorations/ is part of the architecture, not a convenience folder. It gives prototype work an explicit home outside the canonical pipeline.

Why This Is Not a Literal Clone

The site is intentionally similar to Clo-Author at the workflow level, but not at the implementation label level. The manual should help a Codex user operate the system correctly, not cosmetically.
