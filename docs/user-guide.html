User Guide

How to operate the Dittonomics workflow in practice

How It Works

Dittonomics keeps the original research workflow logic:

  • the main Codex session acts as orchestrator
  • narrow subagents handle bounded specialist roles
  • creators create, critics review
  • the workflow advances only when artifacts clear their quality gates

What changes is the implementation surface. Instead of relying on Claude-specific configuration layers, the Codex adaptation uses:

  • AGENTS.md and repo overlays
  • Codex skills with explicit kernel references
  • Codex custom subagents in TOML
  • explicit workflow references
  • optional voice files for prose-sensitive writing and review

Research Workflow

Phase 1: Discovery

Discovery should produce:

  • a research specification
  • a literature map
  • a data assessment

The relevant roles are:

  • librarian and librarian_critic
  • explorer and explorer_critic

The usual explicit entry point is $clo-discover.

Phase 2: Strategy

Strategy locks the estimand, design, assumptions, falsification logic, and robustness plan before coding starts.

The relevant roles are:

  • strategist
  • strategist_critic

The usual explicit entry point is $clo-strategize.

Phase 3: Execution

Execution translates the design into:

  • cleaned data products
  • analysis scripts
  • tables and figures
  • writer handoff artifacts

The relevant roles are:

  • data_engineer
  • coder
  • coder_critic

The active default language stance in this adaptation is:

  • Python primary
  • Stata co-default
  • R and Julia supported when the repo or the user explicitly centers them

Phase 4: Writing

Writing is now documented more explicitly as its own orchestrated layer.

The relevant roles are:

  • writer
  • writer_critic

The public writing stack now emphasizes:

  • section contracts
  • paragraph planning before drafting
  • design-aware causal language
  • humanizer after structure is sound
  • optional voice hierarchy only when the user wants it

The usual explicit entry point is $clo-write.

Phase 5: Peer Review

Peer review is not a grammar pass. It is where the workflow asks whether the project is credible, scoped correctly, journal-calibrated, and ready to move forward.

The relevant roles are:

  • editor
  • domain_referee
  • methods_referee

The usual explicit entry point is $clo-review.

Phase 6: Submission

Submission requires more than polished prose. The workflow should verify:

  • replication readiness
  • journal fit
  • final package completeness

The relevant role is:

  • verifier

The usual explicit entry point is $clo-submit.

Shared Contracts

The newer public starter assumes a stronger contract between the orchestrator and its skills.

In practice that means:

  • the orchestrator resolves target paths before dispatch
  • agents receive a task packet instead of relying on loose prose instructions
  • artifacts have names, roles, and freshness expectations
  • critics and referees return reports to the parent instead of deciding where reports live

Quality Gates

The active scoring thresholds remain:

  • >= 80: commit-ready
  • >= 90: merge-ready
  • >= 95 with no weak component: submission-ready

These thresholds matter because the architecture assumes critique is operational, not decorative.

Voice Workflow

Voice is optional. If you do not care about prose consistency beyond normal editing, you can ignore it entirely.

If you do care, the public pattern is:

  • keep voice files in ~/voice
  • treat them as style guidance, not as truth about evidence
  • keep structure, causality, and section logic governed by the relevant workflow skill
  • use optional overlays for collaboration or journal tightening instead of rewriting the core voice every time

Exploration Workflow

explorations/ remains part of the public workflow because exploratory work exists in real research and should have an explicit home.

Use explorations/ for:

  • prototypes
  • side investigations
  • speculative identification checks
  • draft data or measurement experiments

Do not use it for:

  • canonical cleaned data
  • manuscript-ready figures or tables
  • the final source of truth for analysis code

Minimum rule

If exploratory work is needed and explorations/ does not exist, create:

  • explorations/README.md
  • explorations/ARCHIVE/

Then create a subfolder per exploration.

Graduation rule

Once an exploration becomes part of the real pipeline, promote the relevant outputs into the repo’s canonical code, output, docs, or manuscript paths and leave a short record of what graduated.

Troubleshooting

The upstream guide says CLAUDE.md

Translate that to:

  • AGENTS.md
  • AGENTS.override.md
  • repo-local skills

The upstream guide mentions .claude/rules

Translate that to:

  • active workflow references
  • source-rule mirrors when provenance matters
  • repo overlays for truly always-on repo behavior

The workflow feels too implicit

Use explicit skill mentions such as $clo-review or $clo-discover instead of relying on automatic inference.

The repo lacks an exploration sandbox

Create explorations/README.md and explorations/ARCHIVE/ before running prototype work.
