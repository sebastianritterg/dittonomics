User Guide

How to run Dittonomics day to day

This page is the real tutorial for the system. If someone on the team asks how to use Dittonomics properly, this is the page to send first.

The Core Mental Model

Most of the time, you do not call agents directly.

You usually call a workflow skill, and the main Codex session decides whether to:

  • do the work itself
  • dispatch a specialist agent
  • dispatch a worker and then a critic
  • ask you for a decision at a real fork

That is the main operating idea:

  • you call skills
  • Codex orchestrates
  • agents stay narrow

The Practical Research Roadmap

If you want the shortest correct way to use Dittonomics, use this sequence:

Research situation Call this first What you should get before moving on
you have a topic, mechanism, or dataset hunch $clo-ideate ranked research questions and a pursue/refine/park/kill screen
you have a candidate question but need grounding $clo-discover literature map, data memo, and better framing
you think the question is real $clo-strategize estimand, design, assumptions, robustness plan
the design is ready to become code $clo-analyze cleaned pipeline, outputs, and reproducible handoff
the evidence is ready to narrate $clo-write section drafts with proper structure and restraint
the draft needs serious criticism $clo-review findings, referee-style risks, editor calibration
real referee or editor reports arrive $clo-revise roadmap, validated task graph, execution blocks, response package
the paper is nearly done $clo-submit verification and submission-readiness check

This is the main tutorial logic of the site. If a colleague asks, “What do I use now?”, the answer should usually come from this table first.

The Daily Work Loop

Morning: resume cleanly

Start with:

Use $clo-research-tools resume-context for this repo and reconstruct where we left off, the important files, unresolved issues, and the best next action.

What this is meant to do:

  • read the latest checkpoint in .codex-state/
  • compare it to current git state
  • remind you what mattered yesterday
  • suggest the next action instead of making you rebuild context manually

During the day: use the workflow skills

Use the skill that matches the real task.

The easiest rule is:

  • if the question is still fuzzy, go left in the pipeline and use ideation
  • if you are already coding, writing, or revising, do not jump backward unless a real flaw forces it
  • if reports arrive from a journal, stop ad hoc edits and move into revision-roadmap mode first

Ideation

Use this before discovery when the project is still just:

  • a topic
  • a cool dataset
  • a mechanism hunch
  • a policy area without a sharp question yet
Use $clo-ideate session on this topic and turn it into ranked research questions with a pursue/refine/park/kill verdict.
Use $clo-ideate rq from this dataset idea and give me the best candidate empirical questions.

Discovery

Use $clo-discover lit on this topic and build the literature map first.
Use $clo-discover data and assess which datasets can support this question.

Strategy

Use $clo-strategize on this question and propose the main design, the key assumptions, and the robustness plan.

Analysis

Use $clo-analyze on this project and focus on building the clean estimation pipeline and the output handoff for writing.

Writing

Use $clo-write intro with current-draft mode and keep econ-intro-writing in charge of structure.
Use $clo-write results and narrate the main tables without overstating causality.

Review

Use $clo-review on this draft and give me the highest-risk findings first.

Revision

Use this when real editor or referee reports already exist.

Use $clo-revise roadmap on these reports and the submitted paper. Extract atomic tasks, validate the dependency logic, and give me execution blocks with go/no-go points.
Use $clo-revise validate quality_reports/revision_tasks_rr1.json
Use $clo-revise execute quality_reports/revision_roadmap_rr1.md and route the approved tasks to the right agents.
Use $clo-revise respond on these reports and the revised draft, using the response matrix and decision log.

Talks and submission

Use $clo-talk seminar to build the deck from the current paper.
Use $clo-submit and check whether this package is actually submission-ready.

End of day: checkpoint intentionally

Use:

Use $clo-research-tools checkpoint for this repo. Capture today's goal, what we finished, blockers, important files, and the best next step tomorrow.

This is the recommended end-of-day ritual.

Explicit Skills vs Natural Language

You can work in either style.

Explicit style

Best when:

  • you want predictable routing
  • you are teaching a colleague
  • you want the website or starter kit to stay easy to follow

Example:

Use $clo-review --peer on the latest draft.

Natural-language style

Best when:

  • the task is obvious
  • you want a conversational workflow
  • you do not care about naming the exact skill

Example:

Please review the current paper draft like a journal referee and focus on identification and contribution.

Recommendation

For daily work with colleagues, prefer explicit $clo-* names.

It makes:

  • the system easier to learn
  • the website cheat sheet easier to use
  • routing behavior easier to remember

How Agent Activation Works

What usually activates agents

What you ask for Typical system response
idea screening or research-question generation ideator then ideator_critic
literature review librarian then librarian_critic
data search explorer then explorer_critic
identification design strategist then strategist_critic
strategic revision roadmap revision_planner then revision_planner_critic
coding or data pipeline work coder or data_engineer, then coder_critic
paper drafting writer
front-end writing econ-intro-writing governs structure, then writing follows that contract
proofreading or writing review writer_critic
peer review editor, domain_referee, methods_referee

When to mention an agent directly

Do it when you want to steer the posture.

Examples:

Use $clo-discover lit, but keep it repo-bibliography-first.
Use $clo-review --peer and make the methods_referee especially tough on the event-study design.

Usually you still call the skill, not the raw agent.

Checkpoint, Resume, And Verify-Edit

These three utilities are the new operational layer.

Checkpoint

Purpose:

  • save the state of a workday
  • leave structured notes for future you or a colleague

Default location:

.codex-state/

Default files:

.codex-state/
  checkpoints/
  latest.json
  latest.md

What gets stored:

  • goal
  • summary
  • blockers
  • open questions
  • important files
  • next step
  • git branch and recent commits
  • directory snapshot

Resume-context

Purpose:

  • pick back up quickly the next day
  • avoid spending the first 10 to 20 minutes reconstructing context

Typical use:

Use $clo-research-tools resume-context for this repo and tell me what I most likely need to do next.

Verify-edit

Purpose:

  • run a manual post-edit safety pass
  • use this instead of hidden background lint hooks

Typical use:

Use $clo-research-tools verify-edit 03_code/src/estimate.py

Use it after:

  • editing Python, R, or Julia scripts
  • changing a manuscript build file
  • touching a critical config file

A Good Team Workflow

For a shared research team, the safest default is:

  1. keep the public starter generic
  2. keep private voice files out of the public repo
  3. use repo-local AGENTS.md for project rules
  4. use checkpoints at real stopping points
  5. keep the website open as the daily cheat sheet

When Colleagues Ask “What Should I Open?”

Recommend this order:

  1. Quick Start for the install split and system overview
  2. User Guide for the actual research roadmap
  3. Reference for daily command lookup
  4. Customization only when they are ready to fork or add a voice layer

That order usually teaches the workflow faster than starting from architecture.

A Good End-Of-Day Prompt

Use $clo-research-tools checkpoint for this repo.

Goal: finish the event-study pipeline and prepare the writing handoff.
Summary: built the cleaned panel, updated the estimation script, and confirmed the main figure compiles.
Blocker: the placebo design still needs one decision.
Important file: 03_code/src/event_study.py
Important file: 04_outputs/figures/main_event_study.pdf
Next step: decide the placebo window and rerun the robustness block before writing.

A Good Next-Morning Prompt

Use $clo-research-tools resume-context for this repo and reconstruct where we left off yesterday, the key files, blockers, and the best next step.

Common Mistakes

Calling the wrong layer

Do not call writer directly unless you have a very specific reason. Usually call $clo-write.

Treating voice as structure

Voice files shape prose. They do not replace identification logic, evidence discipline, or intro architecture.

Forgetting the checkpoint

If you stop work without one, the next session becomes unnecessarily expensive.

Treating references as sole truth

Repo bibliography comes first. Current web or working-paper checks can supplement fast-moving empirical frontiers. Neither should be treated as the only truth source.

Skipping ideation when the question is still vague

Discovery is for formalizing and searching. Ideation is for deciding whether the question deserves that investment.

If the project still sounds more like a direction than a question, start with $clo-ideate.

Jumping into edits before building the revision roadmap

When real referee reports arrive, start with $clo-revise roadmap before rewriting sections or running ad hoc analyses.

That step is what:

  • extracts atomic reviewer tasks
  • shows which requests actually block others
  • flags disagreement and go/no-go decisions early
